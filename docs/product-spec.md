## 2. Краткое описание продукта

`cosmetic-workshop-os` - это локальная рабочая система для специалиста, который самостоятельно производит косметические продукты и работает с индивидуальными рецептурами под клиентов.

Система помогает вести:

- рецепты;
- версии рецептов;
- индивидуальные рецепты клиентов;
- клиентов;
- пожелания клиентов;
- обратную связь;
- заказы;
- компоненты;
- партии компонентов;
- сроки годности;
- тару;
- складские движения;
- производство;
- автоматическое списание;
- себестоимость;
- налог;
- маржу;
- алерты;
- закупочный список;
- импорт и экспорт данных;
- резервные копии;
- onboarding и встроенную помощь.

Главная идея: заменить разрозненные тетради, заметки, ручные расчеты, Excel и зависимость от сторонних сайтов единой локальной системой, которую пользователь может открыть и использовать без интернета.

## 3. Целевой пользователь

Основной пользователь - частный специалист по косметике и кремоварению.

Характеристики пользователя:

- не является техническим специалистом;
- не хочет работать с кодом, GitHub, терминалом и сложными настройками;
- может не любить Excel или пользоваться им ограниченно;
- привык работать с рецептами, компонентами, клиентами и индивидуальными формулами;
- хочет сохранить контроль над своими данными;
- боится потерять доступ к рецептам, если они лежат на чужом сайте;
- хочет работать на своем ноутбуке;
- хочет понимать, что делать в интерфейсе без постоянной помощи разработчика;
- нуждается в подсказках, первом обучении и понятной логике работы.

---

## 4. Главная пользовательская проблема

Пользователь ведет рецепты, компоненты, клиентов, заказы и расчеты в разных местах.

Типовые проблемы:

- рецепты могут храниться на стороннем сайте;
- есть риск потерять доступ к рецептам;
- индивидуальные рецепты клиентов сложно отслеживать;
- изменения формул не всегда сохраняются как версии;
- сложно быстро пересчитать проценты в граммы под нужный объем;
- компоненты и тара могут заканчиваться неожиданно;
- сроки годности партий легко упустить;
- себестоимость и маржа считаются вручную;
- закупочный список формируется вручную;
- история заказов и пожеланий клиентов не связана с рецептурами;
- импорт старых данных из таблиц может быть трудоемким;
- нет единой рабочей среды.

---

## 5. Product vision

Система должна стать личной операционной средой косметической мастерской.

Идеальный пользовательский ответ после внедрения:

```text
Я открываю приложение и вижу, что сегодня нужно сделать,
какие заказы ждут, какие компоненты заканчиваются,
что нужно купить, какие рецепты использовать,
что просил клиент и сколько я заработаю на заказе.
```

## 6. Продуктовое позиционирование

Это не:

- публичный сайт;
- интернет-магазин;
- бухгалтерская программа;
- медицинская система;
- облачный SaaS в MVP;
- мобильное приложение в MVP;
- OCR-система в MVP;
- AI-консультант по рецептурам;
- копия стороннего сайта.

Это:

- локальная учетная система;
- личная база рецептов;
- мини-CRM клиентов;
- склад компонентов и тары;
- производственный журнал;
- калькулятор рецептур и себестоимости;
- система алертов и закупок;
- рабочая среда для индивидуального подхода к клиентам.

## 7. Главные продуктовые принципы

### 7.1. Local-first

MVP должен работать локально на MacBook без обязательного интернета.

Пользователь не должен зависеть от внешнего сайта, сервера или подписки для ежедневной работы.

---

### 7.2. Данные принадлежат пользователю

Рецепты, клиенты, история заказов и производство должны храниться на устройстве пользователя.

Должны быть:

- локальная база данных;
- понятная папка данных;
- backup;
- export;
- инструкция восстановления.

---

### 7.3. Не требовать технических навыков

Пользователь не должен:

- клонировать репозиторий;
- запускать команды в терминале;
- устанавливать Python;
- устанавливать Node.js;
- запускать Docker;
- разбираться с портами и зависимостями.

Пользовательский сценарий:

```text
Скачать пакет
→ открыть приложение
→ пройти первый запуск
→ начать работу
```

---

### 7.4. Не делать техническую админку

Интерфейс должен быть понятным для обычного человека.

Требования:

- человекопонятные названия;
- русские labels для пользователя;
- пустые состояния с объяснением;
- подсказки к сложным полям;
- подтверждения опасных действий;
- понятные ошибки;
- dashboard с ежедневными задачами;
- help center внутри приложения.

---

### 7.5. История важнее перезаписи

Система должна сохранять историю:

- версий рецептов;
- индивидуальных рецептов;
- заказов;
- производства;
- списаний;
- пожеланий клиентов;
- обратной связи;
- импортов;
- backup;
- обновлений.

Нельзя незаметно перезаписывать важные данные.

---

### 7.6. Безопасность данных важнее скорости разработки

Любое действие, которое может изменить важные данные, должно быть:

- явным;
- подтвержденным;
- залогированным;
- откатываемым или хотя бы прослеживаемым.

## 8. Основные пользовательские сценарии MVP

---

### 8.1. Первый запуск

Пользователь открывает приложение первый раз.

Система должна:

- показать welcome screen;
- объяснить назначение приложения;
- предложить папку хранения данных;
- настроить налоговую ставку;
- предложить backup reminder;
- предложить demo data;
- открыть Dashboard;
- показать первые шаги.

Успех сценария:

```text
Пользователь не попадает в пустую непонятную систему.
Он понимает, с чего начать.
```

---

### 8.2. Добавление компонента

Пользователь добавляет компонент, например масло, актив, консервант или воду.

Система должна позволить указать:

- название;
- INCI, если есть;
- категорию;
- роль;
- базовую единицу;
- плотность, если нужна;
- минимальный остаток;
- поставщика;
- заметки.

Успех сценария:

```text
Компонент появился в справочнике и может использоваться в рецептах и закупках.
```

---

### 8.3. Добавление партии компонента

Пользователь добавляет закупленную партию компонента.

Система должна позволить указать:

- компонент;
- количество;
- единицу;
- закупочную цену;
- дату закупки;
- срок годности;
- поставщика;
- номер партии, если есть.

Успех сценария:

```text
Партия учитывается на складе, участвует в себестоимости и сроках годности.
```

---

### 8.4. Добавление тары

Пользователь добавляет банки, флаконы, крышки или другую тару.

Система должна позволить указать:

- название;
- категорию;
- объем;
- единицу;
- себестоимость;
- текущий остаток;
- минимальный остаток.

Успех сценария:

```text
Тара учитывается в заказах, производстве, себестоимости и закупках.
```

---

### 8.5. Создание базового рецепта

Пользователь создает базовый рецепт.

Система должна поддерживать:

- название;
- категорию продукта;
- назначение;
- технологические заметки;
- версии рецепта;
- строки рецепта;
- фазы;
- проценты компонентов;
- проверку суммы процентов.

Успех сценария:

```text
Базовый рецепт сохранен как структурированная формула, а не как обычный текст.
```

---

### 8.6. Расчет рецепта

Пользователь указывает нужный итоговый объем, например 153 г, 287 г или 500 г.

Система должна:

- пересчитать проценты в граммы;
- проверить сумму процентов;
- показать warnings;
- учесть плотность для мл;
- показать примерную себестоимость;
- не скрывать приблизительные расчеты.

Успех сценария:

```text
Пользователь понимает, сколько граммов каждого компонента взять.
```

---

### 8.7. Создание клиента

Пользователь создает карточку клиента.

Система должна хранить:

- имя;
- телефон;
- email, если есть;
- адрес, если нужен;
- аллергии;
- предпочтения;
- особенности;
- заметки.

Успех сценария:

```text
Клиент появляется в базе, и вся его история будет собираться в одной карточке.
```

---

### 8.8. Создание индивидуального рецепта клиента

Пользователь берет базовый рецепт и адаптирует его под конкретного клиента.

Система должна:

- создать копию из базовой версии;
- связать ее с клиентом;
- сохранить причину индивидуализации;
- позволить менять состав;
- не менять исходную базовую версию;
- рассчитывать индивидуальный рецепт отдельно.

Успех сценария:

```text
У клиента есть собственная формула, связанная с историей заказов и пожеланий.
```

---

### 8.9. Фиксация пожелания клиента

Пользователь добавляет пожелание клиента, например:

```text
Сделать крем менее жирным.
Добавить более легкую текстуру.
Уменьшить аромат.
Не использовать конкретный компонент.
```

Система должна:

- сохранить пожелание;
- связать его с клиентом;
- связать с заказом или рецептом, если нужно;
- дать статус;
- позволить отметить пожелание как учтенное.

Успех сценария:

```text
Пожелание не потерялось и может повлиять на следующую версию рецепта.
```

---

### 8.10. Фиксация обратной связи

После заказа пользователь записывает результат:

- понравилось;
- не понравилось;
- реакция кожи;
- что изменить в следующий раз;
- связанный заказ;
- связанный индивидуальный рецепт.

Успех сценария:

```text
Обратная связь становится частью истории клиента и помогает улучшать формулы.
```

---

### 8.11. Создание заказа

Пользователь создает заказ.

Система должна позволить выбрать:

- клиента;
- базовый рецепт или индивидуальный рецепт;
- итоговый объем;
- тару;
- количество;
- цену продажи;
- дату;
- заметки.

Успех сценария:

```text
Заказ связан с клиентом, рецептом, тарой и будущим производством.
```

---

### 8.12. Проверка готовности производства

Пользователь нажимает “Проверить изготовление”.

Система должна проверить:

- есть ли рецепт;
- корректна ли сумма процентов;
- хватает ли компонентов;
- хватает ли тары;
- какие партии будут использованы;
- есть ли просроченные или близкие к сроку годности компоненты;
- есть ли missing density warnings;
- себестоимость;
- налог;
- маржу.

Успех сценария:

```text
Пользователь понимает, можно ли изготовить заказ и что мешает.
```

---

### 8.13. Подтверждение производства

Пользователь подтверждает изготовление заказа.

Система должна:

- показать, что будет списано;
- запросить подтверждение;
- создать производственную партию;
- списать компоненты;
- списать тару;
- обновить статус заказа;
- сохранить себестоимость;
- записать AuditLog.

Успех сценария:

```text
Заказ изготовлен, остатки обновлены, история производства сохранена.
```

---

### 8.14. Алерты

Система показывает предупреждения:

- компонент заканчивается;
- тара заканчивается;
- срок годности скоро истечет;
- срок годности истек;
- для заказа не хватает материалов;
- у рецепта сумма не 100%;
- у компонента не указана плотность;
- пора сделать backup.

Успех сценария:

```text
Пользователь видит проблемы до того, как они сорвут заказ.
```

---

### 8.15. Закупочный список

Система формирует предложения закупки.

Причины:

- остаток ниже минимума;
- не хватает для заказа;
- скоро истечет срок годности;
- прогнозируемая нехватка;
- добавлено вручную.

Успех сценария:

```text
Пользователь понимает, что купить и почему.
```

---

### 8.16. Импорт Excel/CSV

Пользователь загружает таблицу с клиентами, компонентами или остатками.

Система должна:

- создать ImportSource;
- создать ImportDraft;
- дать сопоставить колонки;
- показать preview;
- показать ошибки;
- применить только после подтверждения.

Успех сценария:

```text
Старые данные можно перенести без ручного переписывания и без риска сломать базу.
```

---

### 8.17. Backup/export

Пользователь создает резервную копию или выгружает данные.

Система должна:

- показать, где лежит backup;
- создать файл;
- записать действие в AuditLog;
- дать экспортировать основные данные.

Успех сценария:

```text
Пользователь уверен, что рецепты и клиентская база не потеряются.
```

## 11. Пользовательские роли MVP

В MVP фактически один пользователь:

```text
Owner / мастер
```

Это человек, который:

- создает рецепты;
- ведет клиентов;
- создает заказы;
- подтверждает производство;
- ведет склад;
- делает backup;
- импортирует и экспортирует данные.

Многопользовательские роли не входят в MVP.

Future роли:

```text
assistant
viewer
admin
technical support
```

## 15. UX requirements

### 15.1. Empty states

Плохой вариант:

```text
Нет данных
```

Хороший вариант:

```text
У вас пока нет рецептов.
Начните с базового рецепта, например “Крем дневной”.
[Создать рецепт]
```

Каждый пустой экран должен объяснять следующий шаг.

---

### 15.2. Contextual help

Сложные поля должны иметь подсказки:

- плотность;
- партия;
- минимальный остаток;
- версия рецепта;
- индивидуальный рецепт;
- производственная партия;
- себестоимость;
- маржа;
- срок годности.

---

### 15.3. Confirmation dialogs

Подтверждение обязательно для:

- производства заказа;
- списания остатков;
- применения импорта;
- удаления или архивирования;
- восстановления backup;
- удаления demo data;
- обновления базы.

---

### 15.4. Human-readable errors

Плохо:

```text
ValidationError: decimal parsing failed
```

Хорошо:

```text
В поле “Остаток” нужно указать число. Например: 30 или 30,5.
```

## 16. Data safety requirements

Система должна защищать пользователя от потери данных.

Обязательно:

- база вне папки приложения;
- backup;
- auto-backup перед миграцией;
- AuditLog;
- no silent deletes;
- archive instead of hard delete where possible;
- import preview before apply;
- production confirmation before write-off;
- transaction for production;
- no real data in tests.

---

## 17. Local deployment requirements

MVP должен поставляться как локальный пакет.

Пользователь не должен работать с GitHub напрямую.

Целевой сценарий:

```text
Скачать пакет
→ распаковать
→ открыть приложение
→ пройти первый запуск
→ работать
```

Удаленная установка должна быть возможна через screen sharing.

## 21. Success criteria for MVP

MVP считается успешным, если пользователь может без разработчика:

```text
1. открыть приложение;
2. пройти первый запуск;
3. создать backup;
4. создать компонент;
5. добавить партию компонента;
6. добавить тару;
7. создать базовый рецепт;
8. создать версию рецепта;
9. рассчитать рецепт под нужный объем;
10. увидеть warning по плотности;
11. создать клиента;
12. создать индивидуальный рецепт;
13. добавить пожелание клиента;
14. добавить обратную связь;
15. создать заказ;
16. проверить готовность производства;
17. подтвердить производство;
18. увидеть списание компонентов и тары;
19. увидеть себестоимость, налог и маржу;
20. увидеть алерты;
21. увидеть закупочный список;
22. импортировать таблицу через preview;
23. экспортировать данные;
24. открыть AuditLog;
25. закрыть и снова открыть приложение без потери данных.
```

## 22. Product risks

## 24. Product definition of done

Продуктовая задача считается готовой только если:

